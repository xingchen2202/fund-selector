# Iteration 3 研究发现 — P1-3 经理主理人判定防呆 + P2-1 估值工具失败降级

> 基于 2026-07-24 MCP 底层数据复验

## P1-3 — 经理主理人判定防呆

### 问题现象

MCP 返回的经理顺序**不稳定**，导致主理人判定风险：

| 日期 | MCP 返回的 `基金经理` 字段 |
|------|--------------------------|
| 2026-07-23 | `"郑晓辉 刘睿聪"`（郑在前） |
| 2026-07-24 | `"刘睿聪 郑晓辉"`（刘在前） |

### 根因分析

**实际从业年限**（通过 `get_fund_manager_info` 单独查询）：
- 郑晓辉：~5336 天（~14.6 年）
- 刘睿聪：~1303 天（~3.6 年）
- **郑晓辉才是主理人**

**iteration-2 规则缺陷**：
iteration-2 的 SKILL.md 规则第五条写的是"从业年限最长的标记为主理人"，但：
1. **未明确禁止**以 MCP 返回顺序判定主理人的捷径行为
2. **未要求**输出时附注判定依据（从业天数）
3. 模型可能偷懒：直接取 MCP 返回的第一个名字 → 本次会误判刘睿聪为主理人

### 关键缺陷
- **缺陷 A**：规则缺少"禁止按 MCP 顺序取第一个"的硬性约束
- **缺陷 B**：输出格式未要求附注判定依据（从业天数数值）
- **缺陷 C**：未处理"双经理从业天数接近"的边界情况

### 修复方向
强化经理查询容错规则，增加：
1. 显式禁止按 MCP 顺序判定
2. 必须查询并比较累计从业时间（天数）
3. 输出附注判定依据
4. 增加"差距 < 10% → 双经理制"的边界处理

## P2-1 — 估值工具失败降级

### 问题现象

```
get_valuation_metrics("000300") → {error: "估值指标数据为空 (000300)"}
```

iteration-2 的 SKILL.md 规则要求"调用 get_valuation_metrics 获取沪深300 PE 及历史分位"，但**该工具对沪深300 返回空数据**。

### 根因分析

**iteration-2 规则缺陷**：
规则只写了"调用工具"，未写"调用失败怎么办"。当工具返回空时，模型面临两个选择：
1. 标注"数据不可用" ✅ 正确
2. **编造一个分位数字**（如"约 30% 分位"）❌ 幻觉风险

**工具能力边界**：
`get_valuation_metrics` 是 AKShare 的 `stock_a_indicator` 接口封装，可能：
- 接口本身失效
- 参数格式不匹配（指数代码 vs 股票代码）
- 数据源缺失

### 关键缺陷
- **缺陷 A**：规则缺少工具失败时的降级路径
- **缺陷 B**：未明确"严禁编造估值分位数字"的底线
- **缺陷 C**：未启用备用宏观指标（PMI、M2、北向资金）作为替代判断通道

### 修复方向
在宏观研判铁律中增加估值数据获取的降级链：
1. 优先级 1：`get_valuation_metrics` → 成功则输出
2. 优先级 2：失败 → 改用其他 PE/PB 工具
3. 优先级 3：全部失败 → 标注 [⚠️ 估值数据不可用] + 用 PMI/M2/北向资金定性判断
4. 底线：严禁编造估值分位数字
