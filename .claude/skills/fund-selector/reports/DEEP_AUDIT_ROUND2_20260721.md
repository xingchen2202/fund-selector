## 二、P0 修复验证

### 2.1 约束程序化层（P0-A）✅

**验证方式**：16 个测试，覆盖全部 8 条铁律的边界条件。

| 铁律 | 测试场景 | 结果 |
|------|---------|------|
| 穿透防重叠 | 双基金 20% 重合 → 失败；8% 重合 → 通过 | ✅ |
| 穿透防重叠 | 三基金复杂穿透（A-B 高、B-C 边界） | ✅ |
| 预算硬平衡 | 定投 > 月储蓄 → 失败 | ✅ |
| 预算硬平衡 | 加倍定投 + 无闲置现金 → 失败 | ✅ |
| 预算硬平衡 | 加倍定投 + 有闲置现金 → 警告（标注来源） | ✅ |
| 配置闭环 | 偏差 3% → 通过；偏差 8% → 失败 | ✅ |
| 常识校验 | PE=80x 极端值 → 警告 | ✅ |
| 财务预检 | 无应急金/高息负债/无保险 → 警告 | ✅ |
| 费率穿透 | 费率缺失 → 失败；多基金逐个检出 | ✅ |
| 再平衡 | 未设定 → 警告 | ✅ |
| 免责声明 | 缺失 → 警告 | ✅ |
| 边界 | 空基金列表、缺失大量字段 → 不崩溃 | ✅ |
| 综合 | 全约束合规 → 完全通过无警告 | ✅ |

**结论**：8 条铁律已从"文档指令"升级为"程序化强制"。约束校验器对边界条件处理健壮。

**遗留改进点**（P2）：
- 当前为"事后校验"，未来可改为"生成中约束"（推荐生成时即规避违规）
- 缺少行业穿透的**自动修复建议**（"建议替换为 XX 行业基金"）

### 2.2 综合器风险否决（P0-B）✅

**验证方式**：5 个高冲突场景。

| 场景 | 旧行为 | 新行为 | 结果 |
|------|--------|--------|------|
| 成长5+风险1+价值4+周期3 | 首推 3.2 星（风险被稀释）❌ | top_pick=None，否决 ✅ | ✅ |
| 单基金多视角1星（风险+周期） | 首推 ❌ | top_pick=None ✅ | ✅ |
| 多基金：A 被否决，B 正常 | — | 首推 B ✅ | ✅ |
| 全部≥2星 | 首推最高 | 首推最高（无否决）✅ | ✅ |
| 最低2星（非1星） | — | 不否决 ✅ | ✅ |

**结论**：风险否决机制工作正常。1 星 = 直接否决，2 星 = 不否决，边界清晰。

**遗留改进点**（P2）：
- 当前否决是"一票否决"，未来可改为**加权否决**（2 个视角 2 星 > 1 个视角 1 星）
- 可增加"否决申诉"机制（标注"虽被否决但 XX 理由可考虑"）

## 六、新发现的问题（本轮）

### 🟡 中等（建议修复）

1. **约束校验器缺乏自动修复建议**：检出违规后只说"失败"，未建议"如何修复"（如"制造业重合 20%，建议将基金B 替换为科技/消费行业基金"）

2. **综合器否决后无备选方案**：全部否决时 top_pick=None，但未提供"最接近合规的候选"或"降低风险后的替代方案"

3. **约束校验器未集成到工作流**：当前为独立工具，skill 定义未强制要求调用。建议：在 SKILL.md 增加"推荐生成后必须调用 constraint_validator"的强制要求

### 🟢 轻微（可选修复）

4. **三情景估值表格对齐**：`three-scenario` 输出的表格列宽不固定（显示问题，不影响功能）
